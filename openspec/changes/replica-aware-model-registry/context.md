# Context: replica-aware-model-registry

## Mechanism notes

- **Bus namespace**: `NAMESPACE_MODEL_REGISTRY = "model_registry"` on the existing `cache_invalidation` version-counter table polled every 0.5s by `CacheInvalidationPoller`. Ordering is strictly **write-then-bump**: the snapshot row commit happens before the version bump, mirroring the api_key/firewall namespaces, so a follower that wakes on the bump always finds the new payload.
- **Content-hash skip**: the leader hashes the canonical JSON payload (sha256, `sort_keys` + compact separators, frozensets serialized as sorted lists). If the hash equals the last persisted/applied hash, the payload write and the bump are skipped; only `refreshed_at`/`leader_id` are touched (guarded by `WHERE content_hash = :hash`, falling back to a full upsert if the row changed underneath). This prevents bump storms and multi-MB rewrites for stable catalogs while keeping snapshot age meaningful.
- **SQLite vs PostgreSQL upsert atomicity**: PostgreSQL uses `INSERT .. ON CONFLICT (id) DO UPDATE` (row-level lock, atomic). SQLite uses the same statement form, atomic under SQLite's single-writer lock. While the SQLite everyone-is-leader bypass exists (owned by the harden-scheduler-leader-election sibling), concurrent writers are last-writer-wins over content derived from the same upstream, hence idempotent.
- **Cleared marker**: a registry clear (no active accounts) persists `{"cleared": true, ...}` rather than deleting the row, so followers distinguish "leader intentionally reverted to bootstrap floor" from "no snapshot ever persisted".
- **Staleness cap**: `model_registry_snapshot_max_age_seconds` (default 86400). Snapshots whose `refreshed_at` is older are ignored everywhere the store is read (startup, poller callback, tick backstop); the bootstrap catalog remains the floor and the next leader refresh repopulates. Because the leader touches `refreshed_at` every refresh tick even for unchanged content, a healthy leader keeps the snapshot perpetually fresh.
- **Rolling-deploy skew**: `schema_version` (currently 1) is stored beside the payload. A mismatch is logged at warning and the snapshot ignored — never an error. Old-code replicas never read the table; a new-code follower behind an old-code leader simply behaves as before this change. Any change to the payload shape MUST bump `SCHEMA_VERSION` in `app/core/openai/model_registry_store.py`, and any new `UpstreamModel`/`ModelRegistrySnapshot` field MUST be added to the codec (the field-complete round-trip unit test fails otherwise).
- **Ops**: table `model_registry_snapshot` (single row id=1) is safe to delete at rest; replicas fall back to the bootstrap catalog until the next leader refresh. `leader_id` records which instance last wrote, for diagnostics only. Persist logs payload size; reconcile logs applied model counts.

## Example

Replicas A (leader) and B on PostgreSQL. Upstream withdraws `o4-legacy` and gates `gpt-5.9-pro-only` to Pro plans. A's refresh tick updates its in-memory registry, persists the snapshot, and bumps `model_registry`. Within ~0.5s B's poller sees the version change, loads the payload, imports it under the registry lock, and invalidates its local account-selection cache. B's `/v1/models` now matches A's; B rejects `o4-legacy` (`is_suppressed_model` true) and routes `gpt-5.9-pro-only` only to Pro accounts. If the bump had been lost, B's next non-leader refresh tick (<=300s) notices the header mismatch and reconciles anyway — without ever fetching the upstream catalog itself.
